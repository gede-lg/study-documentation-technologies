# Overview

**Título:** Domine o `ng generate` do Angular CLI: Guia Completo e Extensivo

---

## 1. Introdução

O comando `ng generate` (ou `ng g`) do Angular CLI é a ferramenta principal para **escalonar** automaticamente classes, componentes, serviços, módulos e muito mais no seu projeto Angular. Com base em *schematics*, ele gera arquivos padronizados e faz as modificações necessárias no código, acelerando o desenvolvimento e mantendo a consistência do projeto ([angular.dev](https://angular.dev/cli/generate)).

## 3. Conceitos Fundamentais

- **Schematic**: roteiro que define como arquivos devem ser criados ou modificados.
- **Coleção de Schematics**: agrupamento de vários schematics (ex.: `@schematics/angular`).
- **Escopo do Comando**: o `ng generate` delega ao schematic escolhido (ex.: `component`, `service`, `module`, etc.) as regras de criação e integração no projeto.
- **Angular DevKit**: infraestrutura por trás dos schematics, responsável por ler templates, aplicar transformações e atualizar o *workspace* Angular.

---

## 4. Sintaxe e Uso

```bash
ng generate <schematic> [nome] [--opções]
# ou, versão abreviada:
ng g <schematic> [nome] [--opções]

```

- `<schematic>`: tipo de artefato a gerar (component, service, module etc.).
- `[nome]`: identificador/nome do artefato (sem extensão).
- `[--opções]`: parâmetros que controlam detalhes de geração (ver seção “Opções”).

### Exemplos rápidos

```bash
# Gera um componente 'login' dentro de src/app/login
ng g component login

# Gera um serviço 'auth' e atualiza providers
ng g service auth

# Gera um módulo 'user' sem criar pasta extra
ng g module user --flat

```

---

## 5. Argumentos

| Argumento | Descrição | Tipo |
| --- | --- | --- |
| `schematic` | Nome do schematic a ser executado (por exemplo, `component`, `service`) | string |
| `nome` | Nome do artefato a ser criado (sempre sem extensão) | string |

> Observação: para listar todos os schematics disponíveis, você pode usar ng generate --help, que exibirá também os sub-comandos suportados (angular.dev).
> 

---

## 6. Opções

| Opção | Alias | Tipo | Padrão | Descrição |
| --- | --- | --- | --- | --- |
| `--defaults` |  | boolean | false | Desativa prompts interativos para opções que já têm valor padrão. |
| `--dry-run` | `-d` | boolean | false | Simula a geração, mostrando o que seria feito sem alterar arquivos. |
| `--force` |  | boolean | false | Sobrescreve arquivos existentes sem pedir confirmação. |
| `--help` |  | boolean | — | Exibe a mensagem de ajuda para o comando `generate`. |
| `--interactive` |  | boolean | true | Habilita ou desabilita prompts interativos durante a execução. |

> Para cada schematic há ainda options específicas (por exemplo, --module, --flat, --skip-tests no component). Utilize ng g component --help para vê-las em detalhe (angular.dev).
> 

---

## 7. Restrições de Uso

- **Alterações manuais extensas:** não use `ng generate` se o arquivo a ser criado exigir lógica ou estrutura muito customizada; prefira criar manualmente.
- **Ambientes de produção críticos:** evite rodar com `-force` em branches importantes sem revisão, para não sobrescrever código sensível.
- **Integração com templates legados:** schematics podem entrar em conflito se o projeto tiver estrutura fora do padrão Angular.
- **Dependência de versões:** verifique sempre a compatibilidade entre sua versão do Angular CLI e o *workspace* (por exemplo, Angular v20 ([en.wikipedia.org](https://en.wikipedia.org/wiki/Angular_%28web_framework%29?utm_source=chatgpt.com))).

---

## 8. Melhores Práticas e Casos de Uso

1. **Use `-dry-run` antes de aplicar** para pré-visualizar mudanças.
2. **Modularize via schematics**: gere módulos e componentes agrupados (`ng g module feature/user --routing` + `ng g component feature/user/list`).
3. **Personalize defaults** no `angular.json` para dispensar flags frequentes (ex.: sempre gerar SCSS, não gerar testes).
4. **Commit antes de gerar**: facilita reverter caso algo não saia como esperado.
5. **Aproveite `-skip-tests`** em protótipos rápidos.
6. **Crie seus próprios schematics** para padrões repetitivos do time.

## 9. Exemplos Completos

**Contexto:** gerar um módulo `products` com componente de listagem e serviço de API.

```bash
# 1. Gera o módulo com roteamento
ng g module products --routing --flat

# 2. Gera componente ProductsList dentro do módulo
ng g component products/products-list --export

# 3. Gera serviço de consumo de API
ng g service products/products-api

```

**Resultado esperado no `app-routing.module.ts`:**

```
const routes: Routes = [
  { path: 'products', loadChildren: () =>
      import('./products/products.module').then(m => m.ProductsModule)
  }
];

```

**Estrutura de pastas:**

```
src/app/
├── products/
│   ├── products.module.ts
│   ├── products-routing.module.ts
│   ├── products-list/
│   │   ├── products-list.component.ts
│   │   └── products-list.component.html
│   └── products-api.service.ts

```
