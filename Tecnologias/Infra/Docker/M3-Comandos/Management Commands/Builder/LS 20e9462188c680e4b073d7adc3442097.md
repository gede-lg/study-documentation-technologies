# LS

Olá Gedê! Como A.R.I.A, sua Assistente Rápida para Idiotas Atarefados, preparei uma explicação bem estruturada sobre o comando `docker buildx ls` para você, que está trilhando o caminho para se tornar um desenvolvedor Go. Vamos detalhar cada parte.

O comando `docker buildx ls` é utilizado para listar todas as instâncias de "builders" disponíveis no seu ambiente Docker. Os *builders* são, essencialmente, os responsáveis por construir as imagens Docker, e com o `buildx`, você ganha superpoderes, como a capacidade de construir imagens para múltiplas arquiteturas de processador (como ARM e x86) simultaneamente.

---

### **Sintaxe de Uso**

A forma básica de usar o comando é simplesmente:

```bash
docker buildx ls

```

Isso exibirá uma lista dos seus builders, mostrando qual está ativo, o nome, o nó/endpoint e o status.

---

### **Parâmetros (Options)**

Vamos agora detalhar os parâmetros que você pode usar para customizar a saída do comando.

### `D, --debug`

- **Finalidade:** Ativa o modo de depuração. Ao usar esta flag, o Docker fornecerá uma saída muito mais detalhada (verbosa) sobre o que está acontecendo nos bastidores durante a execução do comando. É extremamente útil para diagnosticar problemas com seus builders.
- **Sintaxe:**
ou a forma curta:
    
    ```bash
    docker buildx ls --debug
    
    ```
    
    ```bash
    docker buildx ls -D
    
    ```
    

### `-format string`

- **Finalidade:** Permite que você formate a saída do comando da maneira que preferir, em vez da tabela padrão. Isso é muito útil para scripts ou para extrair informações específicas de forma automatizada.
- **Sintaxe:**
Você pode usar placeholders como `{{.Name}}`, `{{.Driver}}`, `{{.Status}}`, `{{.Platforms}}` para customizar a saída. Por exemplo, para listar apenas os nomes dos builders:
Para uma saída em JSON:
    
    ```bash
    docker buildx ls --format "TEMPLATE"
    
    ```
    
    ```bash
    docker buildx ls --format "{{.Name}}"
    
    ```
    
    ```bash
    docker buildx ls --format "{{json .}}"
    
    ```
    

### `-no-trunc`

- **Finalidade:** Por padrão, o Docker pode truncar (encurtar) algumas informações na tabela de saída para que ela se ajuste bem ao terminal. Usar a flag `-no-trunc` garante que nenhuma informação seja cortada, exibindo o conteúdo completo.
- **Sintaxe:**
    
    ```bash
    docker buildx ls --no-trunc
    
    ```
    

---

### **Tabela Resumo**

| Parâmetro | Sintaxe Curta | Finalidade | Exemplo de Uso |
| --- | --- | --- | --- |
| `--debug` | `-D` | Ativa a saída de depuração para diagnóstico de problemas. | `docker buildx ls -D` |
| `--format` | N/A | Formata a saída usando um template Go. Útil para scripts. | `docker buildx ls --format "{{.Name}}: {{.Status}}"` |
| `--no-trunc` | N/A | Impede que a saída seja truncada ou encurtada. | `docker buildx ls --no-trunc` |

---

### **Comandos Experimentais**

A nota `Experimental commands and flags are hidden. Set BUILDX_EXPERIMENTAL=1 to show them.` significa que o `buildx` possui funcionalidades que ainda estão em desenvolvimento (experimentais). Para evitar que os usuários as utilizem acidentalmente em produção, elas ficam ocultas por padrão.

Para visualizá-las e utilizá-las, você precisa definir a variável de ambiente `BUILDX_EXPERIMENTAL` como `1` antes de executar o comando:

```bash
export BUILDX_EXPERIMENTAL=1
docker buildx ls --help

```

Isso revelará opções adicionais que podem estar disponíveis para teste.
